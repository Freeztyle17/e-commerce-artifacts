## 1. Контекст и границы

### 1.1 Описание системы

**Название:** Система управления складом для e-commerce (WMS-lite)  

**Назначение:** Лёгкая система класса WMS, предназначенная для автоматизации складских процессов интернет-магазина. Обеспечивает приёмку товаров, размещение, обработку заказов (комплектация, упаковка, отгрузка), управление остатками в реальном времени и интеграцию с платформой e-commerce.

WMS-lite заменяет Excel-таблицы, бумажные журналы и разрозненные инструменты единой системой с мобильными терминалами сбора данных (ТСД), штрих-код/QR-сканированием и автоматической синхронизацией остатков.

**Типы обрабатываемых процессов:**
-	Приёмка и размещение товаров от поставщиков
-	Комплектация и упаковка клиентских заказов
-	Отгрузка и формирование документов для служб доставки
-	Инвентаризация, пересчёт и корректировка остатков
-	Обработка возвратов и брака
-	Управление адресным хранением (ячейки, зоны, стеллажи)

**Цель системы (миссия):** обеспечить 99 %+ точность остатков, скорость обработки заказов менее 30 минут и полную трассировку каждой единицы товара, чтобы:
-	Остатки мгновенно отражались на сайте e-commerce
-	Складской персонал работал только через мобильные приложения и ТСД.
-	Система работала в offline-режиме с последующей синхронизацией.
-	Минимизировались ошибки комплектации и упаковки.
-	Была полная история всех операций для аудита и аналитики.

### 1.2 Границы системы

**В границы WMS-lite входит:**  
-	Учёт товаров, серийных номеров, сроков годности и партий
-	Адресное хранение и динамическое размещение
-	Генерация и выполнение задач бизнес-процессов
-	Сканирование штрих-кодов / QR / RFID
-	Синхронизация остатков и заказов в реальном времени
-	Формирование отгрузочных документов и этикеток
-	Инвентаризация и корректировки
-	Отчёты по складу и аналитика
-	Мобильный интерфейс для складского персонала
	
**В границы WMS-lite (внешние системы) НЕ входит:** 
-	Фронтенд и корзина e-commerce
-	Бухгалтерский и финансовый учёт
-	CRM и база данных клиентов
-	Управление поставщиками и закупками (полностью)
-	Физическая доставка и трекинг посылок
-	Хранение учётных записей сотрудников (корпоративный каталог)
-	Печатающее оборудование и ТСД (только интеграция)

### 1.4 Внешние системы
| Система                    | Направление обмена                                    | Данные                                                      |
|----------------------------|-------------------------------------------------------|-------------------------------------------------------------|
| Платформа e-commerce       | Передаёт заказы, получает остатки и статусы           | Заказы, статусы заказов, остатки                            |
| Службы доставки            | Получают документы, отправляют трекинг-номера         | Накладные, этикетки, трекинг-номера, подтверждения          |
| ERP / 1С Бухгалтерия       | Обмен финансовыми документами                         | Приходные/расходные документы, проводки                     |
| Система поставщиков        | Передаёт данные о приходных партиях                   | Приходные накладные, данные о товарах и партиях             |

| Система «Честный ЗНАК»     | Передаёт коды маркировки, статусы ввода/вывода        | Коды Data Matrix, статусы, отчёты                           |

### 1.5 Ключевые ограничения и допущения

**Ограничения:**  
- WMS-lite не является ERP – не ведёт бухгалтерский учёт  
- Не управляет физическим оборудованием склада (роботы, конвейеры)  
- Не хранит персональные данные клиентов – только обезличенные заказы  
- Не отвечает за физическую доставку  
- Работает по модели «lite» (нет сложных 3PL-функций)  

**Допущения:**  
- Платформа e-commerce предоставляет стабильный API  
- Склад оснащён Wi-Fi и мобильными ТСД со сканерами  
- Корпоративный каталог содержит актуальные учётные записи  
- Службы доставки поддерживают электронный обмен накладными

## 2. Границы Sol

### 2.1 Концепция функционирования (ConOps)

**Система:** Система управления складом для e-commerce (WMS-lite / LogTech) Версия документа: 1.0 Дата: 07.04.2026
Документ описывает, как именно работает WMS-lite в реальных условиях эксплуатации: кто участвует, какие модули задействованы, как протекают ключевые процессы в штатном режиме, при сбоях и в нештатных ситуациях.

### 2.2 Участники

#### Внутренние участники:
- **Менеджер склада / Администратор** – создание задач, настройка зон, дашборды, отчёты, управление интеграциями.  
- **Приёмщик** – приёмка, сканирование, размещение товара.  
- **Комплектовщик** – отбор заказов по маршрутным листам.  
- **Упаковщик** – упаковка, взвешивание, печать этикеток, подготовка к отгрузке.  
- **Складской персонал** – все роли работают через ТСД с авторизацией.

#### Внешние участники (через системы):
-  **Платформа e-commerce**
-  **Службы доставки**
-  **Поставщики**
-  **ERP / 1С Бухгалтерия**
-  **Корпоративный каталог сотрудников**
-  **Система маркировки «Честный ЗНАК»**

### 2.3. Системные элементы (типовая конфигурация)

####WMS-lite состоит из следующих ключевых модулей:
-  **Модуль приёмки и размещения** — обработка приходных документов, сканирование, размещение по ячейкам (в том числе без распоряжения).
-  **Модуль адресного хранения** — управление зонами, стеллажами, ячейками, динамическое размещение, учёт по партиям, характеристикам и срокам годности (FEFO).
-  **Модуль комплектации и упаковки** — генерация маршрутных листов, задач на отбор и упаковку, сканирование, взвешивание.
-  **Модуль отгрузки** — формирование накладных, этикеток, передача документов службам доставки.
-  **Модуль инвентаризации** — быстрый пересчёт без остановки работы склада, мгновенная синхронизация с 1С.
-  **Модуль работы с маркировкой «Честный ЗНАК»** — распознавание, фиксация и передача кодов маркировки, статусы ввода/вывода из оборота.
-  **Мобильный интерфейс + интеграция с ТСД** — Android/iOS-приложение для складского персонала, работа в offline-режиме с последующей синхронизацией.
-  **Журнал аудита и трассировка** — полная история всех операций с указанием исполнителя (авторизация по ролям).
-  **Интеграционный шлюз** — двунаправленная синхронизация с e-commerce, ERP/1С, службами доставки и системой «Честный ЗНАК».

### 2.4. Сценарии использования

---

#### 2.4.1 Сценарий №1: Приёмка товаров от поставщика (по распоряжению)
**Цель:** Принять партию товаров, проверить, разместить по ячейкам и мгновенно обновить остатки на сайте e-commerce.

**Предусловие:**
-  Получена приходная накладная из 1С или от поставщика.
-  Менеджер склада создал задачу на приёмку.
-  Приёмщик авторизован в системе через ТСД.
  
**Сценарий:**

 1. Приёмщик получает задачу на ТСД.
 2. Сканирует штрих-код приходной накладной.
 3. Сканирует каждый товар (включая коды «Честный ЗНАК»).
 4. Система проверяет соответствие количества и характеристик.
 5. Приёмщик размещает товар в предложенную ячейку (адресное хранение).
 6. Система фиксирует размещение и отправляет данные в ERP/1С и e-commerce.
 7. Приходная партия появляется в системе с актуальными остатками.

**Постусловие:** Товар размещён, остатки обновлены в реальном времени, вся операция зафиксирована в журнале аудита.

**Исключения:**
-  Несоответствие количества / брак — система предлагает создать акт расхождения.
-  Код «Честный ЗНАК» не читается — переход в ручной режим с фотофиксацией.
-  Нет связи с 1С — операция сохраняется локально и синхронизируется позже.

**Альтернативный сценарий:**
-  Offline-режим ТСД: все данные сохраняются локально до восстановления связи.
-  Частичная приёмка: система позволяет завершить операцию по уже отсканированным товарам.

---

#### 2.4.2 Сценарий №2. Комплектация и упаковка клиентского заказа
**Цель:** Собрать заказ клиента за минимальное время (< 30 минут) с нулевой ошибкой комплектации.

**Предусловие:**
-  Поступил новый заказ из платформы e-commerce.
-  Создана задача на picking + packing.
-  Остатки товара подтверждены.
  
**Сценарий:**

1. Комплектовщик получает маршрутный лист задач на ТСД.
2. Система подсказывает оптимальный маршрут по складу (FEFO).
3. Комплектовщик сканирует каждую позицию в заказе.
4. После комплектации задача автоматически передаётся упаковщику.
5. Упаковщик сканирует товары, взвешивает посылку, печатает этикетки и штрих-коды «Честный ЗНАК».
6. Подтверждает готовность — система формирует отгрузочные документы.

**Постусловие:** Заказ полностью собран, упакован, готов к отгрузке, остатки списаны, статус передан в e-commerce.

**Исключения:**
-  Товара нет в ячейке — система автоматически ставит заказ на удерживание и уведомляет менеджера.
-  Повреждение товара при упаковке — создание заявки на возврат/брак.
-  Режимы деградации:
-  Нет сети — ТСД работает offline, задача синхронизируется при восстановлении связи.
-  Проблемы с принтером этикеток — печать через резервный принтер или отложенная печать.

---

#### 2.4.3 Сценарий №3. Отгрузка и передача заказа в службу доставки
**Цель:** Передать готовый заказ службе доставки с полным комплектом документов и трекинг-номером.

**Предусловие:**
-  Заказ упакован и подтверждён.
-  Служба доставки доступна через API.

**Сценарий:**

1. Упаковщик подтверждает готовность на ТСД.
2. Система формирует накладную и этикетку.
3. Передаёт данные в службу доставки.
4. Получает трекинг-номер и обновляет статус заказа.
5. Данные автоматически уходят в ERP/1С и e-commerce.

**Постусловие:** Заказ передан в доставку, клиент видит трекинг на сайте.

**Исключения:**
-  Служба доставки не принимает документ — автоматический переход на резервную службу.
-  Ошибка в весе/габаритах — возврат на перепаковку.

**Альтернативный сценарий:**
- Нет связи со службой доставки — документы сохраняются и отправляются позже (отложенная отгрузка).

---

#### 2.4.4 Сценарий №4. Инвентаризация без остановки склада
**Цель:** Провести пересчёт товаров без остановки приёмки/отгрузки и мгновенно скорректировать остатки.

**Предусловие:**
-  Менеджер склада запустил задачу инвентаризации по выбранным зонам/товарам.

**Сценарий:**

1. Система разбивает задачу на небольшие подзадачи по ТСД.
2. Складской персонал сканирует товары в процессе обычной работы.
3. Система сравнивает фактические остатки с учётными.
4. При расхождении автоматически создаётся акт корректировки.
5. Данные мгновенно синхронизируются с 1С и e-commerce.

**Постусловие:** 
-  Актуальные остатки, акт корректировки, полная история пересчёта.
 
**Альтернативный сценарий:**
-  Полная работа в offline-режиме с последующей сверкой.

---

#### 2.4.5 Сценарий №5. Обработка возвратов и брака
**Цель:** принять возвращённый клиентом товар, проверить его состояние, принять на склад или списать в брак с корректным обновлением остатков и документов для e‑commerce и 1С.

**Предусловие:**
-  Поступил возврат от клиента через e‑commerce (сформирована заявка на возврат).
-  Менеджер склада создал задачу на приёмку возврата.
-  Сотрудник по возвратам авторизован в ТСД.

**Сценарий:**
1. Сотрудник получает задачу на ТСД.
2. Сканирует штрих‑код возвратной накладной или QR‑код из заявки.
3. Сканирует каждую товарную позицию (включая коды «Честный ЗНАК»).
4. Система сверяет состав возврата с заказом.
5. Сотрудник выбирает состояние товара: «годен к продаже», «брак», «уценка».
6. При выборе «годен» – система предлагает ячейку для размещения и восстанавливает остаток на сайте.
7. При выборе «брак» – система списывает товар по акту брака и отправляет данные в 1С.
8. Все операции фиксируются в журнале аудита с фотофиксацией брака (опционально).

**Постусловие:** Возврат обработан, остатки скорректированы, статус возврата передан в e‑commerce и 1С.

**Исключения:**
-  Несоответствие возвращённого товара заявке – создание расхождения, уведомление менеджера.
-  Отсутствие кода «Честный ЗНАК» на бракованном товаре – ручной ввод + фотофиксация.
-  Возврат без документа – создание внеплановой задачи на приёмку.

**Режимы деградации:**
-  Offline‑режим ТСД – все операции сохраняются локально, синхронизация при восстановлении связи.
-  Нет связи с e‑commerce – возврат обрабатывается, но остатки на сайте корректируются позже (с пометкой «ожидает синхронизации»).

## 4.2 Ключевые архитектурные решения

---

### 4.2.1 **Решение №1. Монолитное ядро с логическим разделением на модули + offline-first мобильные клиенты**

**Контекст:** WMS-lite предназначена для средних и небольших интернет-магазинов (от 50 до 500 заказов в день). Команда разработки — небольшая или средняя (3–8 человек). Критически важны: скорость обработки заказов (менее 30 минут), работа склада при нестабильном Wi-Fi и высокая точность остатков (99%+). Полноценная микросервисная архитектура будет избыточной на данном этапе.

**Решение:** Серверная часть реализована как единое монолитное приложение с чётким логическим разделением на модули (приёмка, комплектация, адресное хранение, интеграционный шлюз и др.). Модули взаимодействуют через внутренние вызовы. Мобильные клиенты (ТСД) работают по принципу offline-first с локальным хранилищем и очередью изменений.

**Отвергнутая альтернатива:** Полноценная микросервисная архитектура с независимым развёртыванием каждого модуля (Kubernetes + сервисная сетка).

**Trade-off:** Простота разработки и высокая скорость vs ограниченное масштабирование модулей.

**Сравнение**
|Критерий	               | Монолит + модули               |	Микросервисы         |
|--------------------------|--------------------------------|------------------------|
| Сложность разработки	   | Низкая	                        | Высокая 				 |
| Скорость разработки	   | Высокая	                    | Средняя/низкая		 |
| Задержки между модулями  | Минимальные (внутри процесса)	| Сетевые + сериализация |
| Offline-поддержка на ТСД | Простая и надёжная	        	| Сложная				 |
| Сложность развёртывания  | Низкая							| Высокая				 |
| Отказоустойчивость       | Средняя						| Высокая				 |

**Что получаем:**
1.	Высокую скорость работы системы — критично для комплектации заказов менее чем за 30 минут.
2.	Простоту и скорость разработки и отладки — один процесс, один лог, одна точка мониторинга.
3.	Надёжный offline-режим на терминалах сбора данных без сложной синхронизации между сервисами.
4.	Низкий порог входа для команды — не требуется глубокая экспертиза в распределённых системах.

**Что теряем:**
1.	Невозможность независимо масштабировать отдельные модули (например, только модуль Честный ЗНАК при пиковой нагрузке).
2.	Сбой в одном модуле (например, в интеграционном шлюзе) может повлиять на работу всего сервера.

**Почему принимаем компромисс:**
1.	При текущих и прогнозируемых объёмах (до 1000–1500 заказов в сутки) горизонтальное масштабирование всего монолита (несколько экземпляров за балансировщиком) полностью покрывает потребности.
2.	Чёткое логическое разделение модулей с хорошо определёнными интерфейсами сохраняет возможность поэтапного перехода к микросервисам в будущем без тотальной переработки.
3.	Мониторинг здоровья каждого модуля в интеграционном шлюзе снижают риски каскадных сбоев.

---

### 4.2.2 **Решение №2. Обязательное сканирование + криптографическая защита журнала аудита**

**Контекст:** Система должна обеспечивать 99%+ точность остатков и полную трассируемость каждой единицы товара. Операции на складе должны иметь юридическую значимость при проверках «Честный ЗНАК», возвратах и внутренних аудитах.

**Решение:** Все ключевые операции (приёмка, размещение, отбор, упаковка, инвентаризация) требуют обязательного сканирования штрих-кода / QR / Честный ЗНАК. Каждый факт выполнения операции записывается в неизменяемый журнал аудита с криптографической цепочкой хэшей (каждая запись содержит хэш предыдущей).

**Отвергнутая альтернатива:** Использование специализированного аппаратного WORM-хранилища (Write Once Read Many) для журнала аудита.

**Trade-off:** Точность 99%+ и юридическая значимость vs небольшие задержки на сканирование.

**Сравнение**
|Критерий	               | Криптографическая цепочка      | Аппаратное WORM-хранилище  |
|--------------------------|--------------------------------|----------------------------|
| Стоимость           	   | Нулевая (программное решение)	| Высокая 				 	 |
| Гибкость развёртывания   | Любая инфраструктура	        | Привязано к железу		 |
| Скорость записи   	   | Высокая						| Средняя	 				 |
| Обнаружение нарушений    | Автоматическое                 | Физическая невозможность	 |
| Восстановление  		   | Через резервные копии			| Высокая надёжность		 |

**Что получаем:**
1.	Высокую точность учёта и защиту от человеческого фактора.
2.	Полную историю операций с привязкой к сотруднику, времени и устройству.
3.	Юридическую значимость данных для проверок и споров.
4.	Простоту и низкую стоимость реализации.
   
**Что теряем:**
1.	Небольшое увеличение времени выполнения операции (2–4 секунды на сканирование).
2.	Криптографическая цепочка обнаруживает модификацию, но не предотвращает её физически.
   
**Почему принимаем компромисс:**
1.	Обязательное сканирование уже является основным способом достижения 99% точности.
2.	Доступ к журналу аудита сильно ограничен на уровне базы данных.
3.	Регулярные резервные копии + ежедневная проверка целостности цепочки обеспечивают необходимый уровень защиты при разумных затратах.

---

### 4.2.3 Решение №3. Асинхронная обработка интеграций с внешними системами

**Контекст:** Система обменивается данными с e-commerce, 1С, «Честный ЗНАК» и службами доставки. Эти системы могут быть временно недоступны.

**Решение:** Критичные для пользователя операции (сканирование, подтверждение размещения/отбора) выполняются синхронно. Все интеграционные взаимодействия (обновление остатков, передача документов в доставку, отчёты в 1С) идут через асинхронную очередь с механизмами повторов.

**Trade-off:**  Стабильность склада при сбоях vs задержки обновления данных (секунды).

**Что получаем:**
1.	Стабильную работу склада даже при сбоях внешних систем. 

Что теряем: 
1.	Небольшую задержку в обновлении данных во внешних системах (несколько секунд). 

**Почему принимаем:** 
1.	Это критично для выполнения главного требования — склад не должен останавливаться.

# 5. Интерфейсы (ICD-lite)

##5.1 Назначение и общие принципы

Раздел описывает все интерфейсы системы WMS-lite — внешние (с внешними системами) и внутренние (между компонентами самой системы).

**Основные принципы построения интерфейсов:**
-  Обеспечение высокой надёжности и отказоустойчивости
-  Обязательная поддержка offline-режима для мобильных устройств и ТСД
-  Минимизация зависимости от внешних систем в реальном времени
-  Использование стандартизированных форматов данных
-  Полное логирование всех взаимодействий в журнал аудита
-  Версионирование всех интерфейсов
-  Асинхронная обработка неблокирующих операций

##5.2 Внешние интерфейсы

###5.2.1 Интерфейс с платформой e-commerce

**Тип интеграции:** Двунаправленный, в реальном времени.

**Основные типы данных:**

1. Заказ клиента (входящий):
-  Номер заказа
-  Дата и время создания
-  Состав заказа (список позиций: артикул, количество, характеристики)
-  Данные получателя (ФИО, адрес, телефон)
-  Статус заказа

2. Остатки товаров (исходящий):
-  Артикул / SKU
-  Количество доступное к продаже
-  Количество в резерве
-  Срок годности (для товаров с FEFO)
-  Статус (в наличии / заканчивается / отсутствует)

3. Статус заказа (исходящий):
-  Новый статус (собран / упакован / отгружен / удержан)
-  Дата и время изменения статуса
-  Трекинг-номер (после отгрузки)

**Требования:** Обновление остатков после любой операции на складе должно происходить не позднее 3 секунд.

Особенности: Основной канал — API. Резервный канал — обмен файлами (XML).

6.1.	Шкала оценки рисков
Для оценки рисков используется количественная шкала:
Вероятность (P) — насколько вероятно наступление события в течение первого года промышленной эксплуатации:
	1 (Низкая) — менее 20% вероятность

	3 (Высокая) — более 60% вероятность

	1 (Низкое) — незначительное влияние на работу склада
	2 (Среднее) — заметные задержки, финансовые потери, недовольство клиентов

7.3.	Тест-кейсы (основные)
TC-01. Сквозной процесс приёмки товаров 
Цель: проверить полный цикл приёмки от сканирования до обновления остатков. Предусловия: Создана задача на приёмку. 
Ожидаемый результат: Товар размещён, остатки обновлены в e-commerce и 1С, операция зафиксирована в аудите. 
Проверяемые исключения: Расхождение количества, нечитаемый код Честный ЗНАК, offline-режим.
TC-02. Комплектация заказа с FEFO 
Цель: проверить генерацию оптимального маршрута и корректность отбора по срокам годности. 
Ожидаемый результат: Заказ собран без ошибок, остатки списаны корректно.
TC-03. Упаковка и отгрузка заказа 
Цель: проверить процесс упаковки, взвешивания, печати документов и передачи в службу доставки. 
Ожидаемый результат: Получен трекинг-номер, статус заказа обновлён у клиента.
TC-04. Инвентаризация без остановки склада 
Цель: проверить работу фонового пересчёта. 
Ожидаемый результат: Расхождения выявлены, акты созданы, работа склада не прервана.
TC-05. Полная работа в offline-режиме 
Цель: проверить сохранение и последующую синхронизацию данных при отсутствии сети. 
Ожидаемый результат: Все операции успешно синхронизированы без дублей и потерь.
TC-06. Авторизация и разграничение прав доступа 
Цель: проверить корректность работы ролевой модели. 
Ожидаемый результат: Пользователь видит только разрешённые ему задачи и функции.
TC-07. Обработка расхождений и создание актов 
Цель: проверить механизм работы с браком и пересортицей.
TC-08. Нагрузочное тестирование 
Цель: проверить производительность при одновременной работе 20 ТСД и обновлении 100 остатков в минуту.
TC 09. Обработка возврата товара (годен к продаже)
Цель: проверить полный цикл возврата с восстановлением остатка.
Предусловия: Создана заявка на возврат из e commerce.
Шаги: Сотрудник сканирует возврат → выбирает «годен» → размещает в ячейку.
Ожидаемый результат: Остаток на сайте увеличен, в 1С сформирован возвратный документ, операция в аудите.
TC 10. Обработка брака при возврате
Цель: проверить списание брака и создание акта.
Шаги: Сотрудник выбирает «брак» → подтверждает причину.
Ожидаемый результат: Товар списан, акт брака отправлен в 1С, остатки не восстанавливаются.
TC 11. Истечение времени сессии на ТСД
Цель: проверить автоматический выход из системы.
Шаги: авторизоваться на ТСД → не работать 16 минут → попытаться выполнить операцию.
Ожидаемый результат: Система требует повторный вход, текущая задача сохраняется.
TC 12. Масштабирование под нагрузкой
Цель: проверить производительность при пиковой нагрузке.
Предусловия: Запущено 100 виртуальных ТСД, генерируется 1500 заказов в час.
Ожидаемый результат: Время отклика API ≤ 500 мс (95й перцентиль), потерь данных нет.
